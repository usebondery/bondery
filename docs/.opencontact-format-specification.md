### 1. Overview

`.opencontact` is a modern, JSON-based format for storing and exchanging contact information. It is designed to be:

* **Human-readable** — clean JSON with descriptive keys
* **Extensible** — a `custom` object for arbitrary data without polluting the spec
* **Portable** — no internal IDs or vendor-specific fields in the core spec
* **Versioned** — `$schema` and `version` fields enable forward-compatible parsing
* **Multi-value friendly** — arrays for phones, emails, addresses, socials, etc.

The format draws inspiration from the vCard standard (RFC 6350) but replaces the legacy text-based encoding with structured JSON.

***

### 2. Top-Level Structure

Every `.opencontact` file is a JSON object with these top-level keys:

| Key        | Type     | Required | Description                                |
| ---------- | -------- | -------- | ------------------------------------------ |
| `$schema`  | `string` | No       | URL to the JSON Schema for validation.     |
| `version`  | `string` | **Yes**  | Specification version. Currently `"1.0"`.  |
| `contact`  | `object` | **Yes**  | The contact record. See §3.1.              |
| `groups`   | `array`  | No       | Groups this contact belongs to. See §3.11. |
| `metadata` | `object` | No       | Provenance and tracking info. See §3.12.   |

```json
{
  "$schema": "https://opencontact.org/schema/v1.json",
  "version": "1.0",
  "contact": { },
  "groups": [ ],
  "metadata": { }
}
```

### 3. Field Reference

#### 3.1 `contact`

The main contact object.

| Key            | Type       | Required | Description                                                               |
| -------------- | ---------- | -------- | ------------------------------------------------------------------------- |
| `name`         | `object`   | **Yes**  | Structured name. See §3.2.                                                |
| `pronouns`     | `string`   | No       | Preferred pronouns (e.g. `"she/her"`, `"they/them"`).                     |
| `birthday`     | `string`   | No       | ISO 8601 date (`YYYY-MM-DD`). Partial dates allowed: `--MM-DD` (no year). |
| `anniversary`  | `string`   | No       | ISO 8601 date (`YYYY-MM-DD`). Partial dates allowed: `--MM-DD`.           |
| `photo`        | `object`   | No       | Contact photo. See §3.3.                                                  |
| `organization` | `object`   | No       | Employment / org info. See §3.4.                                          |
| `emails`       | `array`    | No       | Email addresses. See §3.5.                                                |
| `phones`       | `array`    | No       | Phone numbers. See §3.6.                                                  |
| `addresses`    | `array`    | No       | Physical addresses. See §3.7.                                             |
| `urls`         | `array`    | No       | Web links. See §3.8.                                                      |
| `socials`      | `array`    | No       | Social media accounts. See §3.9.                                          |
| `notes`        | `string`   | No       | Free-form notes about the contact.                                        |
| `language`     | `string`   | No       | Preferred language. ISO 639-1 code (e.g. `"en"`, `"cs"`, `"de"`).         |
| `timezone`     | `string`   | No       | IANA timezone (e.g. `"Europe/Prague"`, `"America/New_York"`).             |
| `tags`         | `string[]` | No       | Flat labels/categories for the contact.                                   |
| `custom`       | `object`   | No       | Free-form key-value pairs. See §3.10.                                     |

***

#### 3.2 `contact.name`

Structured name of the contact. At least one of the fields must be non-empty.

| Key        | Type     | Required | Description                                                                                       |
| ---------- | -------- | -------- | ------------------------------------------------------------------------------------------------- |
| `prefix`   | `string` | No       | Honorific prefix (e.g. `"Mr."`, `"Dr."`, `"Prof."`).                                              |
| `first`    | `string` | No       | Given / first name.                                                                               |
| `middle`   | `string` | No       | Middle name(s).                                                                                   |
| `last`     | `string` | No       | Family / last name.                                                                               |
| `suffix`   | `string` | No       | Honorific suffix (e.g. `"Jr."`, `"PhD"`, `"III"`).                                                |
| `nickname` | `string` | No       | Informal name / alias.                                                                            |
| `display`  | `string` | No       | Pre-formatted display name. If omitted, consumers should construct one from the component fields. |

**Example:**

```json
{
  "prefix": "Dr.",
  "first": "Jana",
  "middle": "Marie",
  "last": "Nováková",
  "suffix": "PhD",
  "nickname": "Jani",
  "display": "Dr. Jana Nováková"
}
```

**Minimal example (nickname only):**

```json
{
  "nickname": "SunflowerGirl"
}
```

***

#### 3.3 `contact.photo`

A photo or avatar for the contact. Provide either `url` or `base64`, not both.

| Key         | Type     | Required | Description                |
| ----------- | -------- | -------- | -------------------------- |
| `url`       | `string` | No       | HTTPS URL to the image.    |
| `base64`    | `string` | No       | Base64-encoded image data. |
| `mediaType` | `string` | No       | MIME type of the image.    |

**Permitted `mediaType` values:**

| Value           | Description |
| --------------- | ----------- |
| `image/jpeg`    | JPEG image  |
| `image/png`     | PNG image   |
| `image/webp`    | WebP image  |
| `image/gif`     | GIF image   |
| `image/svg+xml` | SVG image   |

**Example (URL):**

```json
{
  "url": "https://example.com/photos/jana.jpg",
  "mediaType": "image/jpeg"
}
```

**Example (embedded):**

```json
{
  "base64": "/9j/4AAQSkZJRgABAQ...",
  "mediaType": "image/jpeg"
}
```

***

#### 3.4 `contact.organization`

Employment or organizational affiliation.

| Key          | Type     | Required | Description                                             |
| ------------ | -------- | -------- | ------------------------------------------------------- |
| `name`       | `string` | No       | Organization / company name.                            |
| `title`      | `string` | No       | Job title (e.g. `"Senior Engineer"`).                   |
| `department` | `string` | No       | Department within the organization.                     |
| `role`       | `string` | No       | Functional role (e.g. `"Lead Developer"`, `"Founder"`). |

**Example:**

```json
{
  "name": "Acme Corp",
  "title": "Senior Engineer",
  "department": "R&D",
  "role": "Tech Lead"
}
```

***

#### 3.5 `contact.emails`

Array of email addresses. Each entry:

| Key       | Type      | Required | Description                                                                        |
| --------- | --------- | -------- | ---------------------------------------------------------------------------------- |
| `value`   | `string`  | **Yes**  | The email address. Must be a valid email format.                                   |
| `type`    | `string`  | No       | Category label. See §5 — Email types.                                              |
| `primary` | `boolean` | No       | `true` if this is the preferred email. At most one per array. Defaults to `false`. |

**Example:**

```json
[
  { "value": "jana@company.cz", "type": "work", "primary": true },
  { "value": "jana.personal@email.com", "type": "personal" }
]
```

***

#### 3.6 `contact.phones`

Array of phone numbers. Each entry:

| Key       | Type      | Required | Description                                                      |
| --------- | --------- | -------- | ---------------------------------------------------------------- |
| `value`   | `string`  | **Yes**  | Phone number. Recommended format: E.164 (`+420123456789`).       |
| `type`    | `string`  | No       | Category label. See §5 — Phone types.                            |
| `primary` | `boolean` | No       | `true` if preferred. At most one per array. Defaults to `false`. |

**Example:**

```json
[
  { "value": "+420 123 456 789", "type": "mobile", "primary": true },
  { "value": "+420 222 333 444", "type": "work" }
]
```

***

#### 3.7 `contact.addresses`

Array of physical/postal addresses. Each entry:

| Key          | Type      | Required | Description                                                      |
| ------------ | --------- | -------- | ---------------------------------------------------------------- |
| `type`       | `string`  | No       | Category label. See §5 — Address types.                          |
| `street`     | `string`  | No       | Street address line 1.                                           |
| `street2`    | `string`  | No       | Street address line 2 (apt, suite, etc.).                        |
| `city`       | `string`  | No       | City or locality.                                                |
| `state`      | `string`  | No       | State, province, or region.                                      |
| `postalCode` | `string`  | No       | Postal / ZIP code.                                               |
| `country`    | `string`  | No       | ISO 3166-1 alpha-2 country code (e.g. `"CZ"`, `"US"`, `"DE"`).   |
| `primary`    | `boolean` | No       | `true` if preferred. At most one per array. Defaults to `false`. |

**Example:**

```json
[
  {
    "type": "home",
    "street": "Vinohradská 42",
    "city": "Prague",
    "state": "Prague",
    "postalCode": "120 00",
    "country": "CZ",
    "primary": true
  }
]
```

***

#### 3.8 `contact.urls`

Array of web links. Each entry:

| Key     | Type     | Required | Description                            |
| ------- | -------- | -------- | -------------------------------------- |
| `value` | `string` | **Yes**  | The URL. Must be a valid absolute URL. |
| `type`  | `string` | No       | Category label. See §5 — URL types.    |

**Example:**

```json
[
  { "value": "https://jananovakova.cz", "type": "website" },
  { "value": "https://linkedin.com/in/jananovakova", "type": "linkedin" },
  { "value": "https://github.com/jananovakova", "type": "github" }
]
```

***

#### 3.9 `contact.socials`

Array of social media accounts. Unlike `urls`, these carry structured platform identification.

| Key        | Type     | Required | Description                                     |
| ---------- | -------- | -------- | ----------------------------------------------- |
| `platform` | `string` | **Yes**  | Platform identifier. See §5 — Social platforms. |
| `handle`   | `string` | **Yes**  | Username or handle on the platform.             |
| `url`      | `string` | No       | Direct profile URL.                             |

**Example:**

```json
[
  { "platform": "twitter", "handle": "@jananovakova", "url": "https://twitter.com/jananovakova" },
  { "platform": "instagram", "handle": "jana.novakova", "url": "https://instagram.com/jana.novakova" },
  { "platform": "mastodon", "handle": "@jana@mastodon.social", "url": "https://mastodon.social/@jana" }
]
```

***

#### 3.10 `contact.custom`

A free-form object for application-specific or user-defined fields. Keys should be `camelCase` strings. Values can be any JSON-serializable type.

**Rules:**

* Keys must be non-empty strings
* Keys must not conflict with any top-level `contact` field name
* Nested objects are permitted but discouraged (keep it flat when possible)

**Example:**

```json
{
  "dietary": "vegetarian",
  "shirtSize": "M",
  "favoriteColor": "#FF6B6B",
  "metAt": "DevConf 2025"
}
```

***

#### 3.11 `groups`

Array of groups/lists the contact belongs to. Groups are lightweight — they carry a name and optional visual metadata.

| Key           | Type     | Required | Description                                                                  |
| ------------- | -------- | -------- | ---------------------------------------------------------------------------- |
| `name`        | `string` | **Yes**  | Group name. Must be non-empty.                                               |
| `description` | `string` | No       | Short description of the group.                                              |
| `color`       | `string` | No       | Hex color for UI display (e.g. `"#FF6B6B"`). Must match `^#[0-9A-Fa-f]{6}$`. |

**Example:**

```json
[
  { "name": "Close Friends", "description": "Inner circle", "color": "#FF6B6B" },
  { "name": "Work Contacts", "color": "#4ECDC4" }
]
```

***

#### 3.12 `metadata`

Optional provenance and tracking information.

| Key             | Type     | Required | Description                                                                                                         |
| --------------- | -------- | -------- | ------------------------------------------------------------------------------------------------------------------- |
| `createdAt`     | `string` | No       | ISO 8601 datetime with timezone (e.g. `"2025-01-15T10:30:00Z"`).                                                    |
| `updatedAt`     | `string` | No       | ISO 8601 datetime with timezone.                                                                                    |
| `source`        | `string` | No       | Name of the application that created the file (e.g. `"bondery"`, `"google-contacts"`).                              |
| `sourceVersion` | `string` | No       | Version of the source application.                                                                                  |
| `uid`           | `string` | No       | Globally unique identifier. Recommended format: `urn:opencontact:<uuid-v4>`. Used for deduplication across imports. |

**Example:**

```json
{
  "createdAt": "2025-01-15T10:30:00Z",
  "updatedAt": "2026-02-23T14:22:00Z",
  "source": "bondery",
  "sourceVersion": "1.2.0",
  "uid": "urn:opencontact:a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

***

### 4. Batch Format

For multiple contacts, use the `.opencontacts` extension (note the plural). The top-level structure changes to:

| Key        | Type     | Required | Description                                                                   |
| ---------- | -------- | -------- | ----------------------------------------------------------------------------- |
| `$schema`  | `string` | No       | URL to the batch JSON Schema.                                                 |
| `version`  | `string` | **Yes**  | Spec version. `"1.0"`.                                                        |
| `contacts` | `array`  | **Yes**  | Array of contact entries. Each entry has `contact`, `groups`, and `metadata`. |

```json
{
  "$schema": "https://opencontact.org/schema/v1-batch.json",
  "version": "1.0",
  "contacts": [
    {
      "contact": { "name": { "first": "Jana", "last": "Nováková" } },
      "groups": [],
      "metadata": {}
    },
    {
      "contact": { "name": { "first": "Petr", "last": "Svoboda" } },
      "groups": [{ "name": "Family" }],
      "metadata": {}
    }
  ]
}
```

### 5. Enumerations & Value Constraints

The following type values are **recommended** but not enforced. Consumers should handle unknown types gracefully.

#### Email Types

| Value      | Description               |
| ---------- | ------------------------- |
| `personal` | Personal email            |
| `work`     | Work / business email     |
| `school`   | School / university email |
| `other`    | Other                     |

#### Phone Types

| Value    | Description         |
| -------- | ------------------- |
| `mobile` | Mobile / cell phone |
| `home`   | Home landline       |
| `work`   | Work phone          |
| `fax`    | Fax number          |
| `pager`  | Pager               |
| `other`  | Other               |

#### Address Types

| Value     | Description                |
| --------- | -------------------------- |
| `home`    | Home / residential address |
| `work`    | Work / office address      |
| `mailing` | Mailing / postal address   |
| `billing` | Billing address            |
| `other`   | Other                      |

#### URL Types

| Value       | Description                 |
| ----------- | --------------------------- |
| `website`   | Personal or company website |
| `blog`      | Blog                        |
| `portfolio` | Portfolio site              |
| `linkedin`  | LinkedIn profile            |
| `github`    | GitHub profile              |
| `gitlab`    | GitLab profile              |
| `other`     | Other                       |

#### Social Platforms

| Value       | Description    |
| ----------- | -------------- |
| `twitter`   | Twitter / X    |
| `instagram` | Instagram      |
| `facebook`  | Facebook       |
| `linkedin`  | LinkedIn       |
| `mastodon`  | Mastodon       |
| `bluesky`   | Bluesky        |
| `threads`   | Threads (Meta) |
| `tiktok`    | TikTok         |
| `youtube`   | YouTube        |
| `snapchat`  | Snapchat       |
| `discord`   | Discord        |
| `telegram`  | Telegram       |
| `whatsapp`  | WhatsApp       |
| `signal`    | Signal         |
| `reddit`    | Reddit         |
| `github`    | GitHub         |
| `gitlab`    | GitLab         |

> Custom platform names are allowed. Use lowercase, alphanumeric identifiers (e.g. `"myplatform"`).

#### Date Formats

| Format       | Example      | Description                                    |
| ------------ | ------------ | ---------------------------------------------- |
| `YYYY-MM-DD` | `2025-05-15` | Full date with year                            |
| `--MM-DD`    | `--05-15`    | Date without year (birthday with unknown year) |

#### Country Codes

Use ISO 3166-1 alpha-2 codes. Examples:

| Code | Country        |
| ---- | -------------- |
| `CZ` | Czech Republic |
| `SK` | Slovakia       |
| `US` | United States  |
| `DE` | Germany        |
| `GB` | United Kingdom |
| `FR` | France         |
| `JP` | Japan          |

#### Language Codes

Use ISO 639-1 two-letter codes. Examples:

| Code | Language |
| ---- | -------- |
| `cs` | Czech    |
| `en` | English  |
| `de` | German   |
| `fr` | French   |
| `es` | Spanish  |
| `ja` | Japanese |

#### Color Format

Hex color string matching the pattern `^#[0-9A-Fa-f]{6}$`.

| Example   | Description |
| --------- | ----------- |
| `#FF6B6B` | Red         |
| `#4ECDC4` | Teal        |
| `#000000` | Black       |

### 7. Minimal Example

The smallest valid `.opencontact` file:

```json
{
  "version": "1.0",
  "contact": {
    "name": {
      "first": "Jana"
    }
  }
}
```

Another minimal example using only a nickname:

```json
{
  "version": "1.0",
  "contact": {
    "name": {
      "nickname": "SunflowerGirl"
    }
  }
}
```

### 8. Comparison with vCard

| Aspect                | vCard (RFC 6350)                       | `.opencontact`                                 |
| --------------------- | -------------------------------------- | ---------------------------------------------- |
| **Encoding**          | Text-based with `;` and `:` delimiters | JSON                                           |
| **Name field**        | `N:Last;First;Middle;Prefix;Suffix`    | Structured `name` object with named keys       |
| **Multi-value**       | `TEL;TYPE=WORK,VOICE:+1-555...`        | `{ "value": "...", "type": "work" }`           |
| **Categories**        | `CATEGORIES:` comma-separated          | Separate `tags` (flat) + `groups` (structured) |
| **Extensibility**     | `X-` prefixed fields                   | `custom` object + `socials` array              |
| **Versioning**        | `VERSION:4.0`                          | `version` + `$schema` for validation           |
| **Provenance**        | `PRODID` (limited)                     | Full `metadata` block                          |
| **Photo**             | Base64 inline in text                  | Separate `url` or `base64` with `mediaType`    |
| **Social media**      | Various `X-` fields (non-standard)     | First-class `socials` array with platform IDs  |
| **Batch support**     | Concatenated files                     | Single JSON with `contacts` array              |
| **Schema validation** | None built-in                          | JSON Schema via `$schema`                      |
